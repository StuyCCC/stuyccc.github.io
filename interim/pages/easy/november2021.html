Problem 23
Solution

The main observation is to realize that if you decrease one of the numbers a, it is optimal to decrease this number as much as possible, before moving onto b. Lets see why our original product is a*b if we decrease a twice we subtract 2*b from our product, but if we decrease a once then b we subtract b+(a-1) from our product, and finally if we decrease b twice we subtract 2*a from our product, This is a then b or b then a is less optimal because the -1 exists, and making our difference less than it could be.

So now we know its better to subtract one completely before moving on to the other. We have two methods to use we completely subtract a, then try to subtract b or completely subtract b, then subtract a. We simulate the answers we get from these two scenarios and then return their minimum.


Problem 22
Solution

Lets define the left point of our answer, l_max, to be the rightmost left point in the given intervals, and the right point of our answer, r_min, to be the leftmost right point in the given intervals. Now notice that for all the given intervals, [l,r], at least one of the two is true: l is in the range [r_min,l_max] or r is.

Here is the proof, by definition r_min <= r, so now there are two cases. If r <= l_max, then r is in the range and our condition is satisfied. If r > l_max, we have to go look at l instead, and there are also two cases here. If l >= r_min then it is trivially in the range, but if l < r_min then we have the case where l < r_min and r > l_max which means [r_min,l_max] is inside of [l,r] which must mean there is a point of [l,r] in [r_min, l_max].

So now we just loop through our input and find these values, and the size of our interval is l_max - r_min. However, if this value is negative that is undefined, and we can just put 0 instead. Try to think about why this works yourself.

Problem 18
Solution

Ok the first observation is that if n and k have different parities (this means that one is odd while the other is even), then the sum is impossible. Lets see why this happens if n is odd and k is even then an even amount of odd numbers will sum to an even number, not an odd one. Similar logic can be seen for n is even and k is odd. In both these cases we output NO.

Next observation is that the smallest sum of k distinct positive odd integers is k^2 (the sum of 1+3+5+...+(2k-1) is k^2). This means that if k^2 > n, then the problem is impossible and we output NO as well.

Now if the two above conditions aren't true the answer is always YES. We can see this through the following: For every query we can make the sequence 1, 3, 5, 7, ..., 2(k-1)-1, rem. Where rem is the remainder after subtracting the sum of the first k-1 terms of the sequences from n. Try to see why this sequences always works yourself, its good practice.

Problem 15
Solution

Notice that the shortest path from 0,0 to a,b is abs(a)+abs(b). So s >= abs(a)+abs(b) for our path to be possible. If s == abs(a)+abs(b) then we don't have to do anything else, but if s > abs(a)+abs(b) we have to make an extra k steps were k = s-abs(a)-abs(b). Notice that whatever amount of steps we use to move out of the shortest path we have to use the same number to move back to the shortest path, (ie if we move k/2 away from the shortest path we have to use k/2 steps to come back). So this means that if k is even we can move the extra steps, but if k is odd we can not. Overall, out final condition becomes if k > 0 and k % 2 == 0 print YES else print NO.

Solution

Since the input is given in strings, you can first use a map to assign each person a unique ID number, so you can work with indices in an array. You are to take in the inputs for all the classes first, and put them in an array indexed based on period number. Next, create an answer array that will store the number of people infected when the i'th period has gone by. You can simulate now, for each class in each period, if there is an already infected person in that class, you can set the rest of the class to infected as well. Problem 12
Solution

The optimal strategy for this problem is to make all the heater have close to the same number of sections. You can see this by realizing (x-1)^2 + 1^2 > (x/2)^2 + (x/2)^2. So now we have "sum" sections and "c" heaters and we need to find the optimal way to split sum into c numbers such that the c numbers are really close together. We can do this by having sum%c heaters with value of sum/c + 1, and having the rest be sum/c. This adds up to sum, and keeps the c heaters close together. Now with this distribution our final anwer becomes (sum%c)*(sum/c+1)*(sum/c+1) + (c-sum%c)*(sum/c)*(sum/c).


Problem 11
Solution

First observation to solve this problem is to realize that all multiples of a and b are painted white (ie anything with the form ax or by, where x,y are integers will be painted white). From there we can realize that anything in the form ax+by will also be painted white, because ax is painted white and ax is by (multiple of b) away from ax+by, so ax+by is also white. Now notice that ax+by is a multiple of gcd(a,b). I won't prove this fact, but just know that any number greater than ab and is a multiple of gcd(a,b) can be written as ax+by (try some examples in your head).

So this means that if gcd(a,b) = 1, then the result would be finite because every number greater than ab would be white, which would leave a finite number of black spots in the range [0,ab]. Now if gcd(a,b) != 1, then there would be numbers greater than ab that aren't painted white, and since we are going off to infinity, this means that there would be an infinite number of black numbers.

Notice we can check the gcd of two numbers using the c++ library function __gcd(a,b). If __gcd(a,b) == 1 print "Finite", else print "Infinite".

Now I'll explain how to check if a number is a fair. To get the digits of the number we can make a copy of it and then repeatedly modulo it by 10 and then divide it by 10 until we get 0.
