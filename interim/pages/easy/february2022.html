First notice N can only be up to 100, so this screams a O(n^4) algorithm. We can use this to brute force all of the corner points of the subrectangle. However after this, we do not have the time limit to loop through another time. This hints some precalculation.

Let there be two grids, the original grid v and another grid b. If v[i][j] >= K, b[i][j] = 1, else set it to 0. Now this turns into a basic 2d prefix sums problem.

A subrectangle is valid if the precalculated sum of the the subrectangle is >= T and the sum for grid b is equal to its area.

Therefore its sum can be precalculated in O(n^2) and check all subrectangles in O(n^4)

Problem 12
Solution

For every query, we are checking each subarray and determining whether it's possible to fill a chosen subarray with the number x with at most y replacements. For each query, the naive approach is to loop through every possible subarray and check if the number of elements in the subarray not equal to x is at most y. However, O(n^2*q) is too slow.

Solution 1: Precomputation with Brute Force

The constraints for N and x are small, but the number of queries is big. This signals some precomputation. Specifically, for every possibility of x we can precompute the answer for y. Let mx[max(x)+1][N] be a 2d array. The first dimension will be for the value of x you are computing, and the second dimension is the number of elements you are replacing, and let it's value be the length of the longest subarray that it satisfies. We can compute the array by looping through every x and every possible subarray. Afterwards for each x, take the prefix max, since the answer for a greater y cannot be smaller than a query with a smaller y with the same x. We can answer queries in O(1) time using mx[x][y]. Complexity is O(n^2*x + q)

Solution 2: Sliding Window

For every query, we can use sliding window to find the longest subarray that has at most y elements that are not equal to x. This is pretty standard. Complexity is O(nq). Implementation in C++ for this solution is here.

Problem 19
Solution

Believe it or not, but this is a graph problem! Hear me out, Let the nodes of the graph be numbers that satisfy the adjacent numbers requirement. We can build the nodes in order and as we go using our favorite graph traversal algorithm: BFS!

Let the current node be x. We can build edges that will connect x to larger valid numbers. Let v be the last digit of x:

if v >= 2: add an edge between x and x*10 + v-2

if v >= 1: add an edge between x and x*10 + v-1

add an edge between x and x * 10 + v

if v <= 8: add an edge between x and x*10 + v+1

if v <= 9: add an edge between x and x*10 + v+2

Now, x is connected to all valid numbers that you can establish from its digits and are bigger. We can use BFS to traverse the nodes in increasing order. First, add numbers 1 to 9 to the queue since they will always be valid. Also, make sure you are adding the edges in order from smallest to greatest.

The N'th node you access is the answer.

Complexity: O(N)
