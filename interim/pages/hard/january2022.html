Problem 06
Solution

The thing to notice about this problem (at least the easy version) is that there aren't that many beautiful numbers, so few we can precompute every possible beautiful number and use that to answer queries.

If k = 1, iterating over all digits 0 to 9 and creating numbers of length 0 to 9 using only that one digit (just a string of 11111 for example). There are very few (< 100) possible numbers so handling this case just involves iterating through all these beautiful numbers and finding the minimum one greater than n.

If k = 2 they can all be generated through iterating over all pairs of digits taken form the digits 0 to 9 and then computing all numbers of length from 1 to 9 using only the two picked digits (if the two digits are 1 and 3 then 133331 is an example number). However as there are considerably more numbers than when k = 1, a little more work has to be done than just iterating over all of them.

By analyzing this code, it runs in no more than O(l * 2^l * d^2) time where l is the maximum length of a number, d is the number of digits to choose from. This runs in time. It is good to note that depending on how you implement generating beautiful numbers you may make a lot of duplicates (I did at least). Filtering out duplicate numbers is a good idea. In the end there should only be around 40,000-41,000 beautiful numbers.

Store all these beautiful numbers in a sorted array (or really any data structure you can easily do a binary search on) and then process queries. As the numbers are in sorted order, finding the smallest beautiful number >= n can be done with a binary search in O(log(# beautiful numbers)) time.

Additionally the editorial on codeforces gives a different solution to this problem and the solution to the harder version of this problem, F2 on this same contest, which is much faster that the one presented here (as it has to deal with a larger possible k).

Problem 10
Solution

The hard thing to notice about this problem is we can find the order letters are removed. It is simply the reverse order they appear in the string. Using this we can create a frequency list for the initial string. For the ith character removed (1 indexed) freq[char i] = frequency of that character in the entire string divided by i.

From then on this problem simply becomes implementation. The first substring of characters to match the frequency list calculated must the original string of characters, if no substring is found, then the answer doesn't exist. Then using this original substring simulate the process of removing characters and evolving the string, we have enough time. After this process is complete, the resultant string should be the same as our input. If so output the computed original string and order. Else output -1.

I think there might be a different way of doing this, not quite sure of it though, this is just what I did.

Problem 12
Solution

This really more of a combo question than a CS question.

Consider a transforming 011 -> 110.

One way to think about this is the third '1' "hopping" over the second '1' to swap with the '0' as described in the problem statement, but more useful is considering the "11" as one unit, call it '2'. Then by the rules of the problem statement we are transforming 011 = 02 -> 110 = 20.

Another, larger example is 100111011 -> 1002102.

Using the rules given in the problem statement any consecutive '0's and '2's can be swapped. Therefore, considering all pairs of consecutive '1's as '2's, the problem can be restated as finding all orientations of some amount of '2's and some amount of '0's.

That is (# of '0's + # of '2's)! / ((# of '0's)! * (# of '2's)!)

The above can be calculated mod some prime using modular inverses found with Fermat's little theorem and a simple factorial function. (remember, a^(p-1) = 1 = a^(-1) (mod p) for prime p and some number a)
